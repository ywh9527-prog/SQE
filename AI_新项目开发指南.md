# AI新项目开发指南

**版本**: v1.0
**创建日期**: 2025-12-06
**适用对象**: 作为项目主操手的AI协作者
**核心目标**: 规范AI主导的项目开发流程，避免后期重构痛苦

---

## 🎯 **AI开发者核心职责**

作为AI主操手，你不仅是代码编写者，更是：
- **架构设计师** - 设计系统整体架构
- **规范制定者** - 建立开发标准和流程
- **质量守护者** - 确保代码质量和可维护性
- **知识管理者** - 创建和维护项目文档

## 📅 **项目开发时间轴与文档创建节点**

### **Day 0: 项目启动 - 架构设计阶段**

#### 🕐 **上午：需求分析与架构设计**

**必须创建的文档：**

##### 1. `PROJECT_ARCHITECTURE.md` (项目架构文档)

- 明确技术选型和架构决策
- 为后续开发提供设计蓝图

## 必须包含：

## 创建时机：

## 🤖 **AI开发者特别注意事项**

### **1. 文档与代码的同步原则**
```markdown
## 黄金法则：
- 改代码必更新文档
- 文档过时等同于代码过时
- 每次提交前检查文档一致性

## 检查清单：
- [ ] 架构变更 → 更新PROJECT_ARCHITECTURE.md
- [ ] 新增模块 → 更新MODULE_STRUCTURE.md和MODULE_INVENTORY.md
- [ ] 接口变更 → 更新相关模块设计文档
- [ ] 规范调整 → 更新DEVELOPMENT_GUIDELINES.md
- [ ] 重要决策 → 记录到DEVELOPMENT_PROGRESS.md
```

### **2. 自我质量检查机制**
```markdown
## 每次代码提交前必须执行：
1. 架构合规性检查
   - 是否遵循分层架构？
   - 依赖关系是否正确？
   - 命名是否符合规范？

2. 代码质量检查
   - 是否有重复代码？
   - 是否有潜在bug？
   - 性能是否有问题？

3. 文档完整性检查
   - 新功能是否有文档说明？
   - 变更是否已记录？
   - 依赖关系是否已更新？

4. 测试覆盖检查
   - 核心功能是否有测试？
   - 边界条件是否考虑？
   - 错误处理是否完善？
```

### **3. 风险预防机制**
```markdown
## 高风险操作预警：
- ⚠️ 修改核心架构 → 必须更新PROJECT_ARCHITECTURE.md
- ⚠️ 调整模块依赖 → 必须更新MODULE_STRUCTURE.md
- ⚠️ 修改公共接口 → 必须通知所有相关模块
- ⚠️ 大规模重构 → 必须制定详细计划并获得确认

## 风险控制流程：
1. 识别风险级别
2. 制定影响范围分析
3. 更新相关文档
4. 创建回滚计划
5. 获得确认后执行
```

## 📋 **AI开发者每日工作流程**

### **🌅 每日开始工作前**
```markdown
1. 查看DEVELOPMENT_PROGRESS.md，了解昨日进度
2. 检查是否有新的架构决策或规范变更
3. 确认今日开发目标和优先级
4. 检查相关模块的依赖状态
```

### **💻 开发过程中**
```markdown
1. 严格遵循DEVELOPMENT_GUIDELINES.md规范
2. 每完成一个功能，立即更新相关文档
3. 遇到技术难点，记录到DEVELOPMENT_PROGRESS.md
4. 定期检查代码质量，及时重构
```

### **🌙 每日结束工作前**
```markdown
1. 更新DEVELOPMENT_PROGRESS.md，记录今日进展
2. 检查代码质量，修复发现的问题
3. 确认所有文档与代码同步
4. 规划明日工作重点
```

### **📊 每周回顾**
```markdown
1. 使用CODE_REVIEW_CHECKLIST.md进行全面检查
2. 更新MODULE_INVENTORY.md，反映最新状态
3. 评估技术债务，制定优化计划
4. 更新QUALITY_REPORT.md，记录质量指标
```

## 🚨 **常见陷阱与解决方案**

### **陷阱1：重代码轻文档**
```markdown
## 问题表现：
- 代码写了很多，文档还是空的
- 架构变更了，文档没更新
- 新协作者看不懂项目结构

## 解决方案：
- 每次修改代码前，先问"需要更新哪个文档？"
- 设置文档完整性检查点
- 将文档更新作为开发完成的必要条件
```

### **陷阱2：架构设计不充分**
```markdown
## 问题表现：
- 开发过程中频繁重构
- 模块间耦合严重
- 新功能添加困难

## 解决方案：
- 项目启动时投入足够时间做架构设计
- 定期审查架构合理性
- 及时发现和解决架构问题
```

### **陷阱3：命名规范不一致**
```markdown
## 问题表现：
- 同一概念有多种命名
- 模块前缀使用混乱
- CSS类名不规范

## 解决方案：
- 在DEVELOPMENT_GUIDELINES.md中明确命名规范
- 使用代码检查工具自动验证
- 定期审查命名一致性
```

## 💡 **给AI开发者的最后建议**

### **记住你的角色**
你不仅是代码编写者，更是项目的守护者。你的每一个决策都影响项目的长期可维护性。

### **保持文档思维**
代码是项目的现在，文档是项目的未来。没有文档的项目终将重构。

### **追求完美但不拘泥**
在规范和现实之间找到平衡，实用主义比理想主义更重要。

### **持续学习和改进**
每个项目都是学习机会，及时总结经验，优化开发流程。
